<objective>
Create a custom local search module for VitePress that enriches the MiniSearch index with frontmatter `tags` field content, so that normal search (without `tag:` prefix) matches documents by their tags.

Purpose: This is Phase 1 of the tag search feature. It establishes the search module framework in `docs/.vitepress/search/` ŌĆö an extensible module structure that Phase 2 (tag-prefix search) and Phase 3 (tag autocomplete) will build upon. For Phase 1, tags are made searchable by injecting them as hidden HTML content through VitePress's documented `_render` customization point. This keeps the implementation minimal while laying the proper architectural foundation.

Output:
- `docs/.vitepress/search/index.ts` ŌĆö the search module exporting `createSearchConfig()` and internal utility types
- Modified `docs/.vitepress/config.mts` ŌĆö updated to import search module and use its configuration
- Enriched MiniSearch index at build output where document tags contribute to searchable tokens
</objective>

<task type="auto">
  <name>Task 1: Create search module at docs/.vitepress/search/index.ts</name>
  <files>docs/.vitepress/search/index.ts</files>
  <action>Create the file `docs/.vitepress/search/index.ts`. This module exports a single `createSearchConfig()` function that returns the search options object to be merged into `themeConfig.search.options` in config.mts. The function encapsulates:

1. **`_render(src, env, md)` function** (per D-01/D-03): This is the core mechanism. It:
   - Calls `await md.renderAsync(src, env)` to render the markdown. After this call resolves, `env.frontmatter` is populated with all YAML frontmatter fields (confirmed by VitePress source at chunk-D3CUZ4fa.js line 40458-40459).
   - Reads `env.frontmatter?.tags`. If it is an array with at least one element, joins the tags with a space and appends `<span style="display:none">${tagsJoined}</span>` to the rendered HTML. This makes tag content part of the HTML that `splitPageIntoSections` will process into the `text` field.
   - If `tags` is absent, empty, or not an array, returns the HTML unchanged.
   - Returns the HTML string.

2. **`miniSearch` search configuration** (per D-06/D-07/D-11): Returns a `miniSearch` object with:
   - `searchOptions`: `{ fuzzy: 0.2, prefix: true, boost: { title: 4, text: 2, titles: 1 } }` ŌĆö the default VitePress settings. The `boost: { text: 2 }` applies to tag content since it resides in the `text` field via injection. Per D-07 and D-11 ("default weight, no special adjustment"), no custom boost coefficient is applied to tags. The implementer may fine-tune per Claude's Discretion.
   - Do NOT include `options` (i.e., `fields` or `storeFields`) ŌĆö the default `['title', 'titles', 'text']` fields are sufficient since tags are injected into `text`.

3. **Module structure for extensibility** (per D-02): The export is a function, not a plain object, so that Phase 2 can add parameters (e.g., enable tag-prefix search mode). The module file itself can be extended with additional exports in later phases without changing the import pattern in config.mts.

**English tag lowercasing** (per D-09): MiniSearch's default `processTerm` already lowercases tokens ŌĆö no custom implementation needed. The `join(' ')` on tags preserves original casing; MiniSearch handles case-insensitive matching at search time.

**Chinese tag handling** (per D-08): MiniSearch's default tokenizer splits CJK text character-by-character, which is sufficient for matching complete Chinese tags like `Ķ┐Éń╗┤` (the two characters `Ķ┐É` and `ń╗┤` will match when searched together as the default `prefix: true` combines with CJK character-level tokenization).

**Special characters** (per D-10): Tags like `docker-compose` are indexed as stored (joined with space). MiniSearch's tokenizer splits on hyphens. Searching `compose` will match the token `compose` from `docker-compose`. Tags like `"@opengsd/gsd-core"` (YAML-quoted) are stored as `@opengsd/gsd-core`. MiniSearch's default tokenizer splits on `/` so `gsd-core` will be a matchable token.

**Do NOT:**
- Inject any UI-affecting HTML (no visible elements, no data attributes for styles, per D-05)
- Modify `fields` or `storeFields` in MiniSearch options (keep defaults, per analysis of VitePress internals)
- Import any external dependencies beyond what VitePress already provides
- Add type definitions that require additional npm packages
